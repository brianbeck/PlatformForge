# Argo CD Manifests

This directory contains Argo CD AppProjects and Application manifests. These files are **generated by Ansible** during `deploy-continuousdeployment.yml` and should be committed to Git after deployment.

## Structure

```
argocd/
├── projects/
│   ├── platform-stage.yml    # AppProject scoping stage applications
│   └── platform-prod.yml     # AppProject scoping prod applications
└── apps/
    ├── stage/
    │   ├── traefik.yml
    │   ├── falco.yml
    │   ├── gatekeeper-core.yml
    │   ├── gatekeeper-templates.yml
    │   ├── gatekeeper-constraints.yml
    │   └── observability.yml
    └── prod/
        └── (same as stage)
```

## How these files are created

1. `ansible-playbook playbooks/deploy-continuousdeployment.yml` runs
2. Jinja2 templates in `ansible/roles/argocd_bootstrap/templates/` are rendered with your repo URL, namespaces, and chart versions
3. Rendered manifests are written to this directory
4. Manifests are applied to the cluster via `kubectl apply`
5. You commit and push so Argo CD can read them from Git

## Why generated instead of static?

These manifests contain environment-specific values:
- Your Git repository URL (for multi-source Applications)
- Namespace names (differ between Model A and Model B)
- Git revision (configurable via `platformforge_repo_revision`)

Generating them from templates avoids hardcoded placeholders like `YOUR_ORG`.

## AppProjects

Each environment gets its own AppProject with scoped permissions:
- **Allowed source repos:** Your PlatformForge repo + Helm chart repos
- **Allowed destinations:** Only platform namespaces + kube-system
- **Cluster resource whitelist:** Only specific types (Namespace, ClusterRole, CRDs, etc.)
- **Namespace resource whitelist:** Only resource types platform services need
- **Prod sync window:** Manual sync required (no auto-sync)

## Applications

Each Application uses Argo CD multi-source to reference:
1. A Helm chart from its upstream repository
2. Values files from this Git repo

Argo CD sync-waves control ordering:
- Wave 0: Traefik, Observability (infrastructure)
- Wave 1: Falco, Gatekeeper core (security)
- Wave 2: Gatekeeper templates
- Wave 3: Gatekeeper constraints

## After deployment

Once services are installed by Ansible and Applications are registered, Argo CD monitors for drift:
- **Stage:** Auto-sync enabled. Push changes to Git and they're applied automatically.
- **Prod:** Manual sync required. Review changes in the Argo CD UI before applying.
